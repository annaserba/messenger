# Что могут спросить на собеседовании

Этот файл помогает защитить проект как system design мессенджера или AI-native платформы. Вопросы сгруппированы по темам, а ответы сформулированы так, чтобы показать инженерные компромиссы, а не просто перечислить технологии.

## Архитектура и границы сервисов

Могут спросить: почему не монолит?

Хороший ответ: для команды 2-3 человек стартовать можно с modular monolith внутри NestJS, но границы сразу выделить по модулям: auth, chat, media, search, ai. Отдельными процессами первыми выносить worker и realtime gateway. Полная микросервисность с первого дня увеличит DevOps-стоимость.

Могут спросить: почему event-driven?

Хороший ответ: отправка сообщения должна завершаться после записи в PostgreSQL и realtime-доставки. AI, push, embeddings, moderation и media processing подписываются на события и не блокируют UX. Это снижает связность и позволяет деградировать частями.

Могут спросить: почему NATS, а не Kafka?

Хороший ответ: NATS JetStream проще для маленькой команды, легче в эксплуатации и хорошо подходит для realtime event bus. Kafka мощнее для огромного durable log и аналитики, но дороже операционно.

Могут спросить: если продукт сейчас маленький, зачем вообще думать о большой архитектуре?

Хороший ответ: думать о росте не значит строить всё сразу. На старте можно делать modular monolith, но заранее держать контракты, domain events, idempotency, cursor pagination, observability и границы модулей. Тогда MVP остаётся простым, а рост не требует переписывать чат-ядро.

Могут спросить: когда реально выносить модуль в отдельный сервис?

Хороший ответ: когда появляется независимая нагрузка, отдельный lifecycle deploy, другие требования к ресурсам или отказоустойчивости. Например, media processing, AI inference и search можно вынести раньше, чем auth/chat core. Выносить сервис только ради “микросервисной архитектуры” не стоит.

## Realtime и доставка сообщений

Могут спросить: как избежать дублей при повторной отправке?

Хороший ответ: клиент генерирует `clientIdempotencyKey`, сервер ставит уникальный индекс `(chat_id, sender_user_id, client_idempotency_key)`. При retry сервер возвращает уже созданное сообщение.

Могут спросить: как масштабировать WebSocket?

Хороший ответ: API-инстансы stateless, Socket.IO rooms распределяются через Redis adapter. JWT валидируется при connect, membership проверяется при `join_chat`, события отправляются в `chat:${chatId}`.

Могут спросить: как гарантировать порядок сообщений?

Хороший ответ: порядок задаёт сервер, не клиент. Для отображения использовать server timestamp/UUIDv7. Если нужно строгое упорядочивание внутри чата, можно добавить `chat_sequence`, выдаваемый транзакционно.

Могут спросить: что делать, если пользователь offline?

Хороший ответ: сообщение сохраняется в PostgreSQL, unread count считается по `last_read_message_id`, push отправляется через worker, при reconnect клиент догружает историю с cursor.

## Данные и поиск

Могут спросить: почему PostgreSQL как основная БД?

Хороший ответ: у мессенджера много связей, транзакций и строгих проверок доступа. PostgreSQL хорошо закрывает users/chats/messages, индексы, полнотекстовый поиск и consistency. Redis/Qdrant/S3 дополняют, но не заменяют source of truth.

Могут спросить: зачем Qdrant, если есть PostgreSQL full-text?

Хороший ответ: full-text ищет слова и формы слов, Qdrant ищет смысл. Запрос “где обсуждали встречу в пятницу” может найти сообщения без точного совпадения слов. Для MVP достаточно PostgreSQL, Qdrant добавляется после.

Могут спросить: как удалять сообщения, если embeddings уже созданы?

Хороший ответ: удаление публикует `message.deleted`; search-service помечает payload `deleted=true` или удаляет point из Qdrant. Legal archive отделён от продуктового поиска.

## AI, приватность и агенты

Могут спросить: почему AI не в MVP?

Хороший ответ: базовое доверие к мессенджеру строится на доставке, приватности и UX. AI должен быть архитектурно предусмотрен, но не должен сорвать сроки первого релиза. Первый AI-кейс после MVP - summary, потому что он понятен пользователю и не ломает коммуникацию.

Могут спросить: как обеспечить privacy-first AI?

Хороший ответ: default `local_only`, явный consent для cloud AI, видимый индикатор cloud processing, запрет cloud fallback в приватных чатах, не логировать raw prompts, E2EE-чаты не индексировать.

Могут спросить: как безопасно дать агенту доступ к чату?

Хороший ответ: агент - участник с ролью `agent`, scoped permissions, audit trail, минимальные права, отдельное разрешение на чтение истории и external tools. Agent output сохраняется как обычное сообщение с `sender_agent_id`.

## Безопасность и законодательство

Могут спросить: как защищаете auth?

Хороший ответ: OAuth state, PKCE для mobile, короткий access token, refresh rotation, Redis session revocation, OTP TTL, rate limit, device/session list.

Могут спросить: как совместить “право на удаление” и обязательное хранение?

Хороший ответ: в продуктовой БД пользовательские данные удаляются или обезличиваются, но юридически обязательные записи уходят в изолированный архив с ограниченным доступом, retention policy и audit.

Могут спросить: почему E2EE сложно?

Хороший ответ: E2EE ломает серверный поиск, AI, moderation и простое восстановление истории. Нужны key management, multi-device protocol, recovery UX и отдельная модель угроз. Поэтому сначала research spike.

## Flutter и мобильный клиент

Могут спросить: как клиент переживает плохой интернет?

Хороший ответ: optimistic UI, локальная очередь исходящих сообщений, idempotency key, reconnect Socket.IO, cursor-based sync после восстановления сети.

Могут спросить: почему Riverpod?

Хороший ответ: Riverpod хорошо подходит для явного состояния, тестируемых notifiers/providers и разделения transport/cache/UI. Для realtime важно контролировать lifecycle подписок.

## Нагрузки и оценка capacity

Могут спросить: сколько выдержит один Node.js инстанс?

Хороший ответ: зависит от payload, heartbeat, CPU и Redis adapter. Не угадывать, а мерить Artillery/k6. Для ориентира планировать десятки тысяч idle WebSocket connections на инстанс и отдельно тестировать fan-out.

Могут спросить: где bottleneck?

Хороший ответ: PostgreSQL writes/indexes, Redis adapter fan-out, push provider latency, S3 upload bandwidth, AI provider latency. AI должен быть вне критического пути.
