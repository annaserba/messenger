# Архитектура AI-native мессенджера

Дата: июнь 2026  
Команда на старте: 2-3 разработчика  
Контекст: приложение начинается как продукт для маленькой компании, но должно иметь путь роста до большой consumer-платформы.  
Цель: простой, надёжный и приватный мессенджер, где AI помогает пользователю, но не ломает базовую доставку сообщений и не забирает приватность по умолчанию.

## 1. Главный архитектурный принцип

Сначала мессенджер должен быть хорошим мессенджером. AI-функции важны, но они не должны находиться в критическом пути отправки и получения сообщений.

Критический путь MVP:

1. Пользователь авторизуется.
2. Открывает список чатов.
3. Отправляет сообщение.
4. Сообщение сохраняется в PostgreSQL.
5. Сообщение доставляется через WebSocket.
6. Остальные процессы запускаются асинхронно через события.

AI, поиск, push, обработка медиа, модерация и аналитика подключаются через event-driven слой. Если AI-сервис недоступен, пользователь всё равно должен отправлять и получать сообщения.

## 2. Высокоуровневая схема компонентов

Компоненты системы:

| Компонент | Роль | Почему выбран |
| --- | --- | --- |
| Flutter client | iOS, Android, web-клиент | Единая кодовая база, быстрый старт, хороший mobile-first UX |
| Local AI runtime | Маленькие локальные модели | Приватные подсказки и короткие задачи без отправки данных в облако |
| NestJS backend | API, auth, chat core, realtime | TypeScript, модульность, зрелая экосистема, удобно для небольшой команды |
| Socket.IO | Realtime-доставка | Быстро запустить, есть reconnect, rooms, Redis adapter |
| PostgreSQL | Основная БД | Транзакции, связи, индексы, полнотекстовый поиск, надёжный source of truth |
| Redis | Сессии, кеш, Socket.IO adapter, очереди | Быстрые ephemeral-данные и горизонтальное масштабирование realtime |
| NATS JetStream | Событийная шина | Легче Kafka, хорошо подходит для realtime event-driven архитектуры |
| S3-compatible storage | Медиафайлы | Отделяет большие файлы от БД, даёт signed URLs и дешёвое хранение |
| Qdrant | Семантический поиск после MVP | Векторный поиск по смыслу, но не нужен в первом релизе |
| Self-hosted AI services | AI-инференс под контролем компании | Данные и стоимость лучше контролируются, чем при полном уходе во внешние API |

Текстовая схема:

Flutter client подключается к NestJS API и Socket.IO gateway. Backend сохраняет данные в PostgreSQL, хранит сессии и realtime-состояние в Redis, складывает медиа в S3-compatible storage и публикует доменные события в NATS JetStream. Workers читают события и выполняют push, медиа-обработку, индексацию, AI-задачи и модерацию. Local AI работает на устройстве для маленьких приватных задач. Self-hosted AI запускается отдельно и вызывается только для задач, где пользователь дал согласие или данные не являются чувствительными.

## 3. Старт маленькой компанией и рост до большой платформы

На старте не нужно строить десятки микросервисов. Это замедлит команду и увеличит DevOps-нагрузку. Правильный старт - modular monolith на NestJS с отдельным worker-процессом.

Стартовая форма:

- один backend-проект;
- модули auth, users, chats, messages, media, realtime, events;
- отдельный worker для очередей и событий;
- PostgreSQL, Redis, NATS, S3-compatible storage;
- Flutter mobile/web client.

Что важно заложить сразу:

| Решение | Зачем сейчас | Как поможет при росте |
| --- | --- | --- |
| Чёткие границы модулей | Не смешивать chat, auth, media и AI | Модули можно вынести в сервисы |
| DTO/contracts package | Клиент и backend используют единые контракты | Проще версионировать API |
| Domain events | AI/search/push не зависят напрямую от chat core | Новые сервисы подключаются подпиской |
| Outbox pattern | События не теряются между БД и NATS | Надёжная доставка при росте нагрузки |
| Idempotency keys | Retry не создаёт дубли | Мобильный клиент безопасно повторяет запросы |
| Stateless API | Инстансы можно масштабировать горизонтально | Load balancer + несколько API-инстансов |
| Cursor pagination | История не грузится целиком | Работает на больших чатах |
| Feature flags | Функции включаются постепенно | Безопасный rollout AI и поиска |
| Observability | Latency и ошибки видны с первого дня | Масштабирование по метрикам, а не на глаз |

Когда выносить модуль в отдельный сервис:

- у модуля отдельная нагрузка;
- модуль требует других ресурсов, например GPU для AI;
- модуль не должен влиять на отправку сообщений при падении;
- появился отдельный lifecycle deploy;
- у модуля отдельные требования безопасности.

Порядок выноса при росте:

1. Worker.
2. Media service.
3. AI orchestrator.
4. Search service.
5. Realtime gateway.
6. Auth service.

Что не делать заранее:

- не начинать с тяжёлой микросервисной архитектуры;
- не вводить Kafka только “на будущее”, если NATS закрывает текущие сценарии;
- не делать service mesh без реальной необходимости;
- не писать свой криптографический протокол;
- не включать AI в путь отправки сообщения.

## 4. Выбор backend-архитектуры

Для маленькой компании лучший старт - NestJS modular monolith. Он проще в разработке, тестировании и деплое, чем набор микросервисов. При этом NestJS позволяет держать границы модулей достаточно строго.

Почему не чистый монолит без границ:

- быстро появляется связность между auth, chat, media и AI;
- сложно выделять команды и зоны ответственности;
- сложно масштабировать отдельные тяжёлые части.

Почему не микросервисы с первого дня:

- много DevOps-стоимости;
- сложнее локальная разработка;
- сложнее транзакции и отладка;
- команда из 2-3 человек будет больше поддерживать инфраструктуру, чем продукт.

Итоговый выбор: modular monolith плюс event-driven integration. Это даёт скорость MVP и путь к росту.

## 5. Выбор realtime-подхода

Socket.IO выбран как прагматичный вариант для MVP:

- поддерживает WebSocket и fallback-механизмы;
- имеет rooms для чатов;
- умеет reconnect;
- хорошо масштабируется через Redis adapter;
- проще для Flutter-клиента, чем ручной WebSocket protocol.

Альтернативы:

| Альтернатива | Почему не первая |
| --- | --- |
| Чистый WebSocket | Больше ручной работы: reconnect, rooms, ack, protocol versioning |
| gRPC streaming | Сложнее для mobile/web realtime UX |
| MQTT | Хорош для IoT, но хуже ложится на обычный chat API |

Для большого роста можно выделить realtime gateway отдельно. Chat core при этом остаётся владельцем правил, прав доступа и записи сообщений.

## 6. Выбор хранения данных

PostgreSQL - source of truth. В нём должны жить пользователи, чаты, участники, сообщения, реакции, настройки, права агентов и metadata медиа.

Почему PostgreSQL:

- нужны транзакции;
- нужны связи между пользователями, чатами и сообщениями;
- нужны индексы и полнотекстовый поиск;
- проще обеспечить консистентность и аудит;
- команда быстрее разработает MVP.

Redis не заменяет PostgreSQL. Он нужен для сессий, presence, typing, краткоживущего кеша, Socket.IO adapter и очередей.

S3-compatible storage не заменяет БД. Оно хранит файлы, а PostgreSQL хранит metadata, права и связи с сообщениями.

Qdrant не должен быть source of truth. Он хранит embeddings и может быть пересоздан из PostgreSQL и event log.

## 7. Модель данных без привязки к коду

Основные сущности:

| Сущность | Назначение |
| --- | --- |
| `users` | Аккаунты, Яндекс ID, телефон, профиль, статус удаления |
| `chats` | Личные, групповые и секретные чаты |
| `chat_participants` | Участники, роли, last read, mute |
| `messages` | Сообщения, sender, тип, текст, статус, soft delete |
| `media_files` | Metadata файлов, ссылки на S3 objects, MIME, размеры |
| `reactions` | Реакции пользователей на сообщения |
| `user_settings` | Приватность, язык, тема, AI consent |
| `blocked_users` | Блокировки |
| `agents` | Описание AI-агентов и capabilities |
| `agent_contexts` | Права и контекст агента внутри конкретного чата |

Индексы, которые важны:

- сообщения по `chat_id` и времени;
- участники по `user_id`;
- полнотекстовый индекс по тексту сообщения;
- уникальность idempotency key для защиты от дублей;
- metadata медиа по `message_id`.

Для большого роста нужно заранее думать о partitioning сообщений по времени или `chat_id`, но не внедрять его до появления реальной нагрузки.

## 8. API и WebSocket-контракты

REST API нужен для:

- авторизации;
- профиля;
- списка чатов;
- истории сообщений;
- загрузки медиа;
- поиска;
- настроек приватности;
- управления агентами после MVP.

WebSocket нужен для:

- подключения пользователя;
- входа в комнату чата;
- отправки сообщения;
- получения нового сообщения;
- typing indicator;
- read receipts;
- presence;
- agent response после MVP.

Выбор: REST для предсказуемых запросов и WebSocket для realtime-событий. Это проще, чем пытаться делать всё через один транспорт.

## 9. Аутентификация

Основной вход - Яндекс ID. Это удобно для российского рынка и снижает сложность хранения паролей. SMS-вход нужен как резервный способ.

Решение:

- access token короткоживущий;
- refresh token с ротацией;
- сессии хранятся в Redis;
- пользователь видит список устройств;
- SMS-коды имеют TTL и rate limit;
- OAuth state и PKCE обязательны для защиты mobile flow.

Почему не пароли:

- больше рисков хранения;
- больше UX-трения;
- выше стоимость поддержки восстановления доступа.

## 10. Event-driven слой

NATS JetStream выбран как основная событийная шина.

События, которые нужны:

- сообщение создано;
- сообщение удалено;
- файл загружен;
- пользователь прочитал чат;
- агент приглашён;
- пользователь изменил AI consent;
- жалоба создана.

Зачем события:

- push не блокирует отправку сообщения;
- AI не блокирует отправку сообщения;
- media processing не блокирует отправку сообщения;
- search indexing можно повторить;
- новые сервисы подключаются без переписывания chat core.

Почему NATS, а не Kafka на старте:

- проще эксплуатация;
- меньше инфраструктурная стоимость;
- быстрее старт для маленькой команды;
- достаточно для realtime domain events.

Kafka можно рассмотреть позже для аналитического event log, если появятся большие объёмы событий и data platform.

## 11. AI-стратегия

AI делится на три уровня:

| Уровень | Где работает | Для чего |
| --- | --- | --- |
| Маленький local AI | На устройстве пользователя | Подсказки, реакции, короткие summary, простая транскрипция |
| Self-hosted AI | На GPU/CPU серверах компании | Суммаризация, embeddings, классификация, транскрипция, агенты |
| Внешний cloud AI | Только с явным согласием | Резерв или сложные задачи, если self-hosted качества не хватает |

Важно: локальный AI должен быть маленьким. Для мобильного устройства нельзя планировать тяжёлую LLM как обязательную зависимость.

Что подходит локально:

- маленькая модель подсказок;
- маленькая embedding/classification модель;
- tiny/small transcription model для коротких голосовых;
- rules + ML hybrid для smart reactions.

Что не стоит делать локально в MVP:

- тяжёлые агентные цепочки;
- длинную суммаризацию больших чатов;
- realtime-перевод голоса;
- сложный RAG по всей истории.

Self-hosted AI - основной путь для контроля стоимости и данных. Для него лучше использовать отдельный AI orchestration layer, а не встраивать AI-вызовы в chat core.

Self-hosted AI-сервисы:

- embeddings service для Qdrant;
- summarization service;
- transcription service;
- moderation/classification service;
- agent runtime;
- model gateway для выбора модели и fallback.

Модели для self-hosted слоя выбираются по задаче:

- маленькие embedding-модели для поиска;
- Whisper-compatible модели для речи;
- Qwen/Llama/DeepSeek-подобные модели для summary, reasoning-задач и агентов;
- отдельные классификаторы для спама и жалоб.

DeepSeek можно использовать, но не как жёсткую зависимость продукта. Его лучше подключать через model gateway, чтобы можно было заменить модель без переписывания AI-оркестратора. Для приватных данных предпочтителен self-hosted или выделенный контур; внешний API допустим только при явном согласии пользователя и понятной политике обработки данных.

## 12. Провайдер инфраструктуры и self-hosted AI

Для старта можно использовать Yandex Cloud, VK Cloud или Timeweb Cloud. С учётом замечания о цене и продуктах Timeweb Cloud стоит рассматривать как практичный вариант, особенно для self-hosted AI и инфраструктуры маленькой компании.

Почему Timeweb Cloud подходит для рассмотрения:

- есть облачные серверы и VDS/VPS;
- есть выделенные серверы;
- есть GPU-серверы для AI-нагрузки;
- есть Managed Kubernetes;
- есть S3-compatible object storage;
- есть облачные базы данных, включая PostgreSQL и Redis;
- есть дата-центры в России, что важно для data residency;
- есть API, CLI и Terraform, что помогает автоматизировать инфраструктуру.

Рекомендуемый подход:

- на MVP держать PostgreSQL, Redis, NATS и backend на обычных облачных/VDS-ресурсах;
- медиа хранить в S3-compatible storage;
- self-hosted AI вынести на отдельный GPU-сервер только после появления реальной AI-нагрузки;
- DeepSeek/Qwen/Llama запускать как отдельный inference service за model gateway, а не внутри chat core;
- не держать GPU постоянно включённым, если задачи редкие и могут выполняться batch-очередью;
- сравнивать Timeweb Cloud, Yandex Cloud и VK Cloud по цене GPU, S3, трафика, managed DB, SLA и юридическим требованиям.

Важно: для персональных данных нужно отдельно проверить договор, регион размещения, соответствие 152-ФЗ, резервное копирование и условия обработки данных.

## 13. RAG и поиск

Для MVP достаточно PostgreSQL full-text search. Это быстрее, дешевле и проще.

Qdrant и RAG стоит добавлять после MVP, когда:

- есть стабильная доставка сообщений;
- есть понятный UX поиска;
- есть политика удаления embeddings;
- есть AI consent;
- есть observability по latency и стоимости.

Почему не сразу Qdrant:

- дополнительная инфраструктура;
- нужно строить embeddings pipeline;
- нужно решать удаление и приватность;
- для первого релиза пользователь больше ценит надёжные сообщения, чем семантический поиск.

Когда Qdrant нужен:

- пользователь ищет не точную фразу, а смысл;
- нужен контекст для summary;
- появляются агенты, которым нужно извлекать релевантные сообщения.

## 14. Multi-Agent поддержка

Агенты - важная часть будущей платформы, но не MVP. На первом этапе достаточно спроектировать модель так, чтобы агент мог быть участником чата позже.

Что заложить заранее:

- `sender_user_id` и `sender_agent_id` как разные источники сообщения;
- роль `agent` среди участников чата;
- permissions на чтение истории, отправку сообщений и external tools;
- audit trail действий агента;
- явный consent пользователя.

Почему A2A не MVP:

- сложная безопасность;
- нужны permissions и sandboxing;
- нужен tool governance;
- высокий риск непредсказуемого поведения;
- ценность для обычного пользователя надо сначала доказать простым агентом.

Первый агент после MVP: Summarizer. Он понятен пользователю, не требует внешних действий и хорошо проверяет AI-инфраструктуру.

## 15. Медиа

Медиа не нужно хранить в PostgreSQL. БД хранит только metadata и связи, файлы хранятся в S3-compatible storage.

MVP:

- изображения;
- документы;
- thumbnails;
- signed URLs;
- проверка MIME и размера;
- базовая защита от опасных файлов.

После MVP:

- видео;
- голосовые;
- транскрипция;
- AI-описания медиа;
- антивирусная проверка на отдельном worker.

Почему так:

- изображения и документы нужны почти сразу;
- видео и голос сложнее по хранению, трафику и обработке;
- транскрипция требует AI pipeline и consent.

## 16. Безопасность и приватность

Базовые решения:

- TLS для HTTPS/WSS;
- короткий access token;
- refresh rotation;
- Redis session revocation;
- rate limit для auth, SMS, send_message и upload;
- блокировки пользователей;
- настройки видимости online status;
- не логировать тела сообщений, OTP, токены и raw AI prompts;
- cloud AI только с явным согласием.

E2EE не стоит делать в MVP. Это отдельная сложная подсистема, которая влияет на поиск, AI, восстановление истории, multi-device и модерацию. Для E2EE нужен отдельный research spike и отдельная модель угроз.

## 18. Функциональные требования

MVP должен позволять:

- войти через Яндекс ID;
- войти по SMS как резерв;
- видеть список чатов;
- создать личный чат;
- создать простую группу;

## 19. Нефункциональные требования

| Категория | Требование | Цель для MVP |
| --- | --- | --- |
| Доставка | Сообщения быстро доходят при активном соединении | p95 до 300 мс внутри региона |
| Надёжность | AI не влияет на отправку сообщений | Chat core работает без AI |
| Консистентность | Retry не создаёт дубли | Idempotency key |
| Масштабирование | API масштабируется горизонтально | Stateless API + Redis adapter |
| Данные | PostgreSQL остаётся source of truth | Redis/Qdrant восстанавливаемы |
| Приватность | Cloud AI только по согласию | `local_only` по умолчанию |
| Безопасность | Auth защищён от brute force | TTL, rate limit, refresh rotation |
| Наблюдаемость | Инцидент можно расследовать | Trace ID, метрики, structured logs |
| Стоимость | AI и медиа контролируются | Очереди, лимиты, self-hosted AI |
| Законодательство | Персональные данные в РФ | RU region для данных и backups |

SLO для MVP:

- `message_delivery_p95_ms <= 300`;
- `send_message_error_rate < 0.5%`;
- `auth_success_p95_ms <= 1000`;
- `websocket_reconnect_p95_seconds <= 5`;
- `push_enqueue_p95_ms <= 500`;
- `postgres_backup_rpo <= 15 минут`;
- `postgres_restore_rto <= 2 часа`.

## 23. Ключевые решения

Итоговые архитектурные выборы:

- стартуем с modular monolith, не с тяжёлых микросервисов;
- chat core не зависит от AI;
- PostgreSQL - source of truth;
- NATS JetStream - event bus;
- Redis - ephemeral state и realtime scaling;
- S3-compatible storage - медиа;
- Qdrant - после MVP;
- local AI - только маленькие модели;
- self-hosted AI - основной путь для контроля данных и стоимости;
- DeepSeek можно использовать через model gateway, желательно в self-hosted или выделенном контуре;
- Timeweb Cloud можно рассматривать как практичный вариант для GPU, S3, Kubernetes и баз данных;
- cloud AI - только с явным согласием;
- A2A, E2EE и mesh - не MVP.
